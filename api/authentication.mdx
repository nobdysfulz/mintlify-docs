Every request to the Knowledgebase API must be authenticated with an API key. Keys are scoped to your workspace and inherit the permissions of the user who created them, so a key created by an Admin can perform any action while a key created by a Viewer is restricted to read-only operations. Keep your keys secret — treat them with the same care you give passwords or private tokens.

## Generating an API Key

You can create as many API keys as you need, and you can revoke them individually at any time. It is good practice to create one key per application or integration so you can rotate or revoke access independently.

<Steps>
  <Step title="Open Workspace Settings">
    Log in to your Knowledgebase dashboard and click your workspace name in the top-left corner, then select **Settings**.
  </Step>
  <Step title="Navigate to the API section">
    In the left sidebar of the Settings page, click **API** under the Integrations section.
  </Step>
  <Step title="Create a new key">
    Click **New API Key**. A dialog will appear prompting you to name the key.
  </Step>
  <Step title="Name the key">
    Enter a descriptive name that identifies where this key will be used — for example, `Production Backend` or `Data Export Script`.
  </Step>
  <Step title="Copy and store your key">
    The full key is displayed once immediately after creation. Copy it to a secure location such as a password manager or secrets manager before closing the dialog.
  </Step>
</Steps>

<Warning>
  Your API key is displayed **only once** at the time of creation. If you lose it, you must revoke the key and generate a new one. Never commit API keys to source control or include them in client-side code.
</Warning>

## Using Your API Key

Pass your API key in the `Authorization` header of every request using the `Bearer` scheme:

```bash
Authorization: Bearer YOUR_API_KEY
```

The following examples show a basic authenticated request to list articles using curl, Python, and JavaScript.

<CodeGroup>

```bash curl
curl https://api.knowledgebase.io/v1/articles \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```python Python
import requests

response = requests.get(
    "https://api.knowledgebase.io/v1/articles",
    headers={"Authorization": "Bearer YOUR_API_KEY"}
)
print(response.json())
```

```javascript JavaScript
const response = await fetch("https://api.knowledgebase.io/v1/articles", {
  headers: {
    "Authorization": "Bearer YOUR_API_KEY"
  }
});
const data = await response.json();
```

</CodeGroup>

## Authentication Errors

When authentication fails, the API returns a `401` or `403` status code with a machine-readable error code in the response body.

| HTTP Status | Error Code | Meaning |
|-------------|------------|---------|
| `401` | `missing_api_key` | No `Authorization` header was included in the request. |
| `401` | `invalid_api_key` | The provided API key does not match any active key in the system. |
| `401` | `revoked_api_key` | The API key was revoked and is no longer valid. |
| `403` | `insufficient_permissions` | The key is valid but the associated role does not permit this action. |
| `403` | `workspace_suspended` | Your workspace has been suspended; contact support. |

A typical `401` error response looks like this:

```json
{
  "error": {
    "code": "invalid_api_key",
    "message": "The API key provided is invalid or has been revoked.",
    "status": 401
  }
}
```

## Key Scopes and Permissions

API keys inherit the role of the workspace member who created them. Knowledgebase has three roles:

| Role | Permissions |
|------|-------------|
| **Admin** | Full read and write access to all resources, including users and workspace settings. |
| **Editor** | Can create, update, and delete articles and collections. Cannot manage users or billing. |
| **Viewer** | Read-only access to published articles and collections. Cannot create or modify content. |

When you build an integration, generate the API key with the least-privileged role required for your use case. For example, a public search widget needs only a Viewer key, while a content publishing pipeline needs an Editor key.

<Tip>
  Rotate your API keys on a regular schedule — every 90 days is a reasonable baseline. Use descriptive key names (e.g. `ci-pipeline-prod`) so you can quickly identify which key belongs to which integration when reviewing your key list.
</Tip>
